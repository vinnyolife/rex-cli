# ContextDB Rehydration After `/new` / `/clear`

- Date: 2026-03-13
- Scope: Docs + wrapper UX (no upstream CLI changes)

## Problem

Users report that after resetting the in-CLI conversation:

- Codex CLI: `/new`
- Claude Code / Gemini CLI: `/clear`

…the next conversation appears to “lose ContextDB memory”.

## Root Cause

AIOS injects ContextDB context packets **only at CLI process start** (wrapper prelude):

`init -> session:latest/new -> context:pack -> start CLI`

In-CLI reset commands wipe the *client’s* conversation state, and the shell wrapper cannot intercept or re-inject context mid-process.

## Solution

1. Expose a stable “rehydration anchor” file on each wrapped start (already generated by `ctx-agent`):
   - `memory/context-db/exports/latest-<agent>-context.md`
2. Document the correct recovery workflow:
   - Preferred: exit CLI and re-run `codex`/`claude`/`gemini`
   - Alternative: in the new chat, ask the agent to read the latest snapshot file (for clients that support `@file` mentions)
3. Make the wrapper print an explicit rehydration hint in interactive mode (so the path is visible before users hit `/new`/`/clear`).
